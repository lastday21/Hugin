# Hugin

Hugin — местный персональный агент поиска работы. Сейчас создана основа серверной части: настройки, фабрика приложения FastAPI и проверка состояния `/health`.

## Требования

- Python 3.12;
- uv 0.11 или новее;
- Docker Desktop с поддержкой Docker Compose;
- Chromium для Playwright (нужен только видимому браузеру Windows).

## Подготовка

```powershell
uv sync --all-groups --all-extras
uv run playwright install chromium
Copy-Item .env.example .env
```

Перед первым запуском замените значение `HUGIN_DATABASE_PASSWORD` в `.env` на длинный случайный пароль. Файл `.env` не попадает в репозиторий.

## Запуск

```powershell
docker compose up -d db --wait
uv run hugin
```

После запуска проверка состояния доступна по адресу `http://127.0.0.1:8000/health`.

При запуске схема PostgreSQL автоматически обновляется до текущей версии. База доступна только с этого компьютера и из внутренней сети Docker.

Ручное управление схемой:

```powershell
uv run hugin-db upgrade
uv run hugin-db current
uv run hugin-db check
uv run hugin-db downgrade
```

Предметная схема поддерживает любое число направлений поиска. У каждого направления свои поисковые запросы, фильтры, правила оценки и набор резюме. Одно резюме можно связать с несколькими направлениями, а одну вакансию — независимо обработать в каждом направлении.

Отклик уникален для сочетания аккаунта hh.ru, вакансии и резюме. Поэтому на одну вакансию можно откликнуться разными резюме, но повторная отправка того же резюме блокируется. Создание намерения отправки и события `APPLY_INTENT` выполняется в одной транзакции.

Очередь хранит приоритет, плановое время, число попыток и отдельное состояние результата. Задания выдаются строго по одному. Состояние `UNKNOWN_RESULT` не разрешает повторную попытку без явного перехода в `RETRY_SCHEDULED`. Пауза, требование входа, CAPTCHA и предупреждение аккаунта относятся к состоянию всей программы и не смешиваются с состоянием отклика.

После анализа все прошедшие жёсткие исключения вакансии автоматически ставятся в очередь для
активного резюме. Сначала выбираются более свежие публикации, а среди них — вакансии с большей
оценкой. Если программа прервалась во время отклика, задание при следующем запуске переводится в
состояние с неподтверждённым результатом: оно не отправляется повторно, а остальные задания
продолжают выполняться.

Схема PostgreSQL также содержит профиль кандидата, подтверждённые факты и разрешения на их
использование, банк ответов, правила компаний, историю обнаружения вакансий, версии инструкций,
индивидуальные письма, версии анкет и ответов, переписку, приглашения, уведомления, происшествия и
сохраняемые настройки. Эти данные разделены так, чтобы подтверждённый текст можно было связать с
его источником и не пересоздавать перед отправкой.

## Запуск в Docker

```powershell
docker compose up -d --build --wait
Invoke-RestMethod http://127.0.0.1:8010/health
```

Docker Compose запускает серверную часть и PostgreSQL 18. Порты приложения и базы доступны только через `127.0.0.1`. Данные PostgreSQL хранятся в томе `hugin-postgres`, остальные рабочие данные — в `hugin-data`. Политика `restart: unless-stopped` поднимает контейнеры после запуска Docker Desktop. Видимый браузер и возможности Windows в контейнер не переносятся.

## Вход в hh.ru

Видимый браузер запускается в Windows, а не внутри Docker. Логин и пароль сохраняются
в защищённом хранилище Windows и не записываются в PostgreSQL, файлы проекта или журнал.
Команды спрашивают пароль скрыто, поэтому он не остаётся в истории PowerShell.

```powershell
uv run hugin-hh save --account-id 1
uv run hugin-hh login --account-id 1
```

Для каждого аккаунта используется отдельный постоянный профиль Chromium в
`%LOCALAPPDATA%\Hugin\browser-profiles`. Если hh.ru запросит одноразовый код или проверку,
программа оставит окно открытым: эти действия выполняются вручную. Удалить сохранённые данные:

```powershell
uv run hugin-hh delete --account-id 1
```

После успешного входа можно загрузить аккаунт и резюме в PostgreSQL:

```powershell
docker compose up -d db --wait
uv run hugin-hh sync --account-id 1
```

Команда сохраняет имя и идентификатор аккаунта, а также названия и идентификаторы резюме.
Повторный запуск обновляет существующие записи без дублей. Резюме, которых больше нет в
аккаунте, остаются в истории базы, но помечаются неактивными. Другие личные данные со страницы
профиля не сохраняются.

## Импорт активного ИТ-резюме

Проверить программное чтение выгрузки hh.ru без записи личных данных в базу:

```powershell
uv run hugin-resume inspect "C:\путь\к\резюме.pdf"
```

После синхронизации аккаунта импортировать одно активное ИТ-резюме:

```powershell
uv run hugin-resume import "C:\путь\к\резюме.pdf" --account-id 1
uv run hugin-resume facts --account-id 1
uv run hugin-resume confirm-fact --account-id 1 --fact-id 1
uv run hugin-resume questions --account-id 1
uv run hugin-resume answer --account-id 1 --key salary_expectation
```

Поддерживаются PDF с текстовым слоем и DOCX, в том числе многостраничные документы и русские
имена файлов. Исходник сохраняется без изменений по контрольной сумме вне репозитория. Извлечённые
факты нельзя использовать до подтверждения. Если частого ответа в резюме нет, он запрашивается
отдельно и после подтверждения сохраняется для следующих анкет.

Если после синхронизации на hh.ru есть несколько резюме с одинаковым названием, при импорте нужно
передать точный идентификатор параметром `--hh-resume-id`. Иначе файл автоматически связывается с
единственным активным резюме на сайте с совпадающим названием.

## Отдельное улучшение резюме

Улучшение не входит в поиск и не меняет исходное резюме. Сначала сохраните ключ и проверенную
облачную модель в защищенном хранилище Windows:

```powershell
uv run hugin-resume-improve configure `
  --folder-id "идентификатор-каталога" `
  --model "aliceai-llm/latest"
uv run hugin-resume-improve test
```

Затем создайте отдельный DOCX-черновик:

```powershell
uv run hugin-resume-improve run `
  --account-id 1 `
  --target-role "Python backend-разработчик"
```

Программа отдельно разбирает места работы и проекты. По каждому блоку она проверяет личный вклад,
результат, стадию, масштаб и взаимодействие, но показывает не больше трех вопросов только по
реальным пробелам. После ответов облачная модель переписывает содержательный блок. Контакты, город,
условия работы, исходный файл и резюме на hh.ru не изменяются. Новый DOCX и отчет сравнения
хранятся в местной папке данных Hugin.

## Поиск вакансий

Настройки поиска хранятся отдельно от резюме. Поисковые запросы можно указать несколько раз,
а города выбрать из встроенного списка. Формат работы, занятость и зарплата без явных параметров
берутся из подтверждённых данных активного ИТ-резюме. Навыки отдельно не вводятся.

```powershell
uv run hugin-hh cities

uv run hugin-hh configure-search `
  --direction "Python backend" `
  --query "Python backend" `
  --query "FastAPI разработчик" `
  --city "Москва" `
  --city "Екатеринбург" `
  --format удалённо `
  --format офис `
  --employment полная `
  --minimum-salary 180000 `
  --desired-salary 220000 `
  --remote-russia

uv run hugin-hh show-search --direction "Python backend"
```

Для города вне встроенного списка используется запись `Название=код_hh`. Города задаются именно
в направлении: место проживания в резюме не ограничивает поиск. Офисные и гибридные вакансии
ищутся в выбранных городах, а удалённые — отдельным поиском по всей России.
Если города не указаны, поиск выполняется по всей России. Если в резюме нет зарплаты, формата,
занятости или другого частого ответа, программа показывает вопрос. Неотвеченный вопрос не
останавливает поиск: соответствующий фильтр просто не применяется.

После настройки команда перебирает все сохранённые запросы и города, использует активное резюме
и сохраняет вакансии без дублей:

```powershell
uv run hugin-hh search --direction "Python backend" --pages 3
```

Для разовой проверки без сохранённой настройки по-прежнему можно передать `--query` и `--area`.
Параметр `--pages` задаёт число последовательных страниц для каждого сочетания запроса и региона;
обход прекращается раньше, если очередная страница пуста.
Поиск использует уже авторизованный профиль браузера. Повторный запуск обновляет вакансии без
дублей и сохраняет, по какому запросу и региону найдена каждая вакансия. Команда не создаёт
отклики и не ставит задания на их отправку.

Загрузить описания найденных вакансий и проверить первые 20 по понятным правилам:

```powershell
uv run hugin-hh analyze --direction "Python backend" --limit 20
```

Для каждой вакансии сохраняются регион, адрес, зарплата, опыт, занятость, формат и график работы,
описание, обязанности, обязательные и желательные требования, навыки, признаки письма, анкеты,
внешней ссылки и проверочного задания, оценка и причины решения. Изменения полей сохраняются в
истории. Результаты делятся на группы «подходит», «с натяжкой» и «отклонена». Почти одинаковые
публикации с разными номерами связываются информационной пометкой, но проверяются и обрабатываются
как самостоятельные вакансии.
Python-разработка с LLM или NLP попадает в группу «с натяжкой», а чистое машинное обучение
отклоняется. Требования от трёх лет, а также пометки «ведущий», `Lead` и `Principal` не являются
запретом. Подходящие вакансии только помечаются для дальнейшего просмотра: эта команда также не
создаёт отклики и задания. Повторный запуск пересчитывает правила по уже загруженным описаниям.

Повторные публикации с другим идентификатором hh.ru сравниваются по компании, названию,
обязанностям и зарплате. Они остаются в истории со ссылкой на основную вакансию, но не попадают в
дальнейшую обработку. Отклонённые вакансии можно отбирать по компании, региону и причине,
открывать как сохранённую карточку и возвращать в обработку:

```powershell
uv run hugin-hh rejected --direction "Python backend" --reason "формат"
uv run hugin-hh vacancy --direction "Python backend" --vacancy-id 123456789
uv run hugin-hh restore --direction "Python backend" --vacancy-id 123456789
```

При возврате сохраняется явное решение пользователя. Следующий пересчёт правил его не отменяет.

## Автоматические отклики

После поиска и проверки вакансий программа может сама подготовить очередь и отправить указанное
число подтверждённых откликов за запуск:

```powershell
uv run hugin-hh apply --direction "Python backend" --limit 5
```

Для направления выбирается активное резюме с наибольшим приоритетом. Сопроводительное письмо
составляется из названия вакансии и технологий, которые одновременно указаны в вакансии и
резюме. Пограничные вакансии включены; отключить их можно параметром `--exclude-stretch`.

Сохранить суточное ограничение и интервал в местной базе:

```powershell
uv run hugin-hh configure-queue --daily-limit 25 --delay-min 30 --delay-max 60
```

Показать очередь, приостановить новые отклики и продолжить работу:

```powershell
uv run hugin-hh queue-status
uv run hugin-hh pause
uv run hugin-hh resume
```

Программа проверяет выбранное резюме, CAPTCHA, состояние входа и предупреждения аккаунта.
Произвольные вопросы работодателя не заполняются выдуманными сведениями: задание переводится в
состояние, требующее ввода пользователя, а остальные вакансии продолжают обрабатываться. Если после
нажатия результат нельзя подтвердить, повторная отправка этой вакансии блокируется до сверки
истории. По умолчанию действует дневное ограничение в 25 подтверждённых откликов и пауза 30–60
секунд между отправками. Настройки и ближайшее разрешённое время следующего отклика сохраняются в
PostgreSQL, поэтому перезапуск не сбрасывает паузу. Дневной счётчик сбрасывается по местному времени
компьютера. Ответ `Retry-After` площадки имеет приоритет над обычным интервалом.

Остановка без удаления рабочих данных:

```powershell
docker compose down
```

## Проверки

```powershell
uv run ruff check .
uv run ruff format --check .
uv run mypy
uv run pytest --cov=hugin --cov-report=term-missing
```

Рабочая база, журналы, профиль браузера, снимки и трассировки не должны находиться в репозитории. В Windows для них по умолчанию используется `%LOCALAPPDATA%\Hugin`.
